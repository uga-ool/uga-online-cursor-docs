What is uga-online-cursor-docs?

This repository is the shared home for UGA Online (OOL) team documentation and Cursor AI configuration. It does not replace the code repos where apps actually live — it sits beside them and tells you (and Cursor) how to work.

Documentation hub

Markdown guides for eLC, React course-file apps, AI agents, Lit components, video/Kaltura, deploy checklists — organized by job, not by mirroring every repo.

Cursor team brain

.cursor/rules/, .cursor/skills/, and .cursor/commands/ apply when this repo (or a multi-root workspace that includes it) is open in Cursor.

Not application code

You generally do not deploy this repo to Brightspace. You edit app code in repos like UGA-Brightspace-React-Apps or uga-lit-components.

💡
Who should clone this? Anyone on the OOL dev/ID team using Cursor on uga-ool GitHub repos. Clone it once next to your other project folders, open it in Cursor, and pull updates when the team adds new rules or docs.
uga-online-cursor-docs → docs + .cursor config → UGA-Brightspace-React-Apps, lit-components, … → eLC / production

GitHub basics (plain language)

If GitHub is new to you, these terms appear everywhere in our workflow. You do not need to memorize Git commands on day one — the Cursor agent can help — but you should understand the ideas.

TermWhat it means for OOL
Repository (repo)A project folder on GitHub. OOL code lives under the uga-ool organization.
CloneCopy a repo from GitHub to your computer so you can edit files. You do this once per repo (or when you get a new machine).
main branchThe primary, shared version of the project. Do not commit directly to main for team code — use a branch and pull request.
BranchYour own line of work (e.g. feature/quiz-export). Safe place to experiment until review.
CommitA saved snapshot of your changes with a message describing why.
PushUpload your commits from your computer to GitHub.
Pull request (PR)Asking the team to merge your branch into main. Reviewers comment on GitHub before merge.
Pull / git pullDownload the latest changes from GitHub to your computer (do this often on main before starting new work).

Typical flow (what you will do)

  1. 1
    Clone the app repo (and this docs repo) to a folder on your Mac/PC.
  2. 2
    Create a branch from main for your task.
  3. 3
    Edit files in Cursor; review AI diffs before accepting.
  4. 4
    Commit & push your branch to GitHub.
  5. 5
    Open a PR on github.com; request review; address comments; merge when approved.
🌐
GitHub in the browser vs terminal You can create PRs and review entirely on github.com. Optional: install GitHub CLI (gh) for commands like gh pr checkout 42. Cursor’s terminal can run git for you when you ask — still review what it runs.

One-time setup on your computer

Follow these steps once. Your team lead can confirm org access and which repos you need.

1. Install tools

  • A
    Cursor — Download and sign in. On first launch, choose Import from VS Code if you already use VS Code (extensions and settings carry over).
  • B
    Git — macOS often has it; Windows: install Git for Windows. Verify: open Terminal and run git --version.
  • C
    GitHub access — Your UGA/GitHub account must be in the uga-ool org. Use HTTPS or SSH to clone (your lead can help with SSH keys).
  • D
    Node.js (for React apps) — Install LTS from nodejs.org if you will run npm install / npm run build.

2. Folder layout (recommended)

Keep sibling folders under one parent (many of us use local_dev/). Example:

local_dev/ uga-online-cursor-docs/ ← this repo (rules + docs) UGA-Brightspace-React-Apps/ ← course-file React monorepo UGA-Brightspace-React-Template/ ← read-only reference uga-lit-components/ uga-online-brightspace-agent-framework/ … other repos you work on

3. Clone this docs repo

# HTTPS example (replace with your path) cd ~/local_dev git clone https://github.com/uga-ool/uga-online-cursor-docs.git

4. Open in Cursor

  1. 1
    File → Open Folder — Select uga-online-cursor-docs to read docs and pick up team rules.
  2. 2
    For app work — Either open the app repo folder alone, or use a multi-root workspace (several repos in one window). Optional examples: workspaces/examples/OOL-Develop.code-workspace — copy to your machine and fix paths; not required if you use your own workspace file.
  3. 3
    Pull updates weekly — In the docs repo: git pull so new rules and guides appear for Cursor automatically.

Daily workflow for OOL developers

A sane default day — adjust with your lead for your role (ID, dev, analytics).

  1. 1
    Open the right repos — Prefer a workspace that includes uga-online-cursor-docs plus the app you’re editing. Avoid having UGA-Brightspace-React-Template in the tree when using Agent mode for feature work (easy to edit the wrong repo).
  2. 2
    Pull latest — git pull on main in each repo you touch, then create your feature branch.
  3. 3
    Ask before Agent — For unfamiliar code or large tasks, use Ask mode to explore; switch to Agent for implementation.
  4. 4
    New React course-file app? — Create under UGA-Brightspace-React-Apps/apps/your-app-name/; ask Cursor to follow scaffold-from-ool-template. Do not build inside the template repo.
  5. 5
    Review every AI diff — Green/red diff in the editor; reject wrong changes.
  6. 6
    Before PR — Run command ool-pre-pr-check; fix flags; run npm run build per app README.

Pull requests & code review

Full checklist for newcomers: docs/cursor/pr-and-code-review.md (markdown, good for printing or sharing).

Authors — before you push

  • ✓
    Changes are in the correct repo (apps under UGA-Brightspace-React-Apps/apps/).
  • ✓
    No .env, keys, or student identifiers in the diff.
  • ✓
    Run ool-pre-pr-check in Cursor.
  • ✓
    PR description includes summary, test plan, and eLC notes (course/OU, role, Manage Files path) when LMS-related.

Reviewers

On GitHub: read description → Files changed → comment or approve. In Cursor: run ool-pr-review with the PR link; use Ask mode to understand unfamiliar files.

## Summary (PR template excerpt) - What changed and why ## Test plan - [ ] npm run build - [ ] Tested in eLC (OU: ___; role: ___; path: ___)

FeatureVS CodeCursor
Editor, terminal, git sidebarYesSame

Agent modes (Ask, Agent, Plan…)

Pick the mode in the chat panel before you prompt. When unsure, start with Ask.

Ask — read-only

Explains code; does not edit files. Safe for learning and PR review prep.

Agent — autonomous

Edits files, runs commands. Always review diffs. Default for implementation.

Plan — approve first

Writes a plan before changing code. Good for large or risky tasks.

Debug

Runtime investigation for bugs. Use when something fails in browser or Node.
🔁
Recommended pattern: Ask (scope) → Plan (optional, big tasks) → Agent (execute) → inline ⌘/Ctrl+K for small text tweaks.

Security & FERPA

Rules in ool-secrets-and-ferpa.mdc apply in every chat. Human judgment still required.

Never commit secrets

No API keys, OAuth secrets, or .env files. Use gitignored env files and institutional secret stores.

FERPA

Do not put student names, UGA IDs, or identifiable grades in prompts or committed files. Use aggregated anonymized data when appropriate.

Privacy Mode

Your org may require Cursor Privacy Mode — prompts not used for model training. Confirm with your admin.
