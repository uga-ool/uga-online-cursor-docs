GitHub basics (plain language)

If GitHub is new to you, these terms appear everywhere in our workflow. You do not need to memorize Git commands on day one — the Cursor agent can help — but you should understand the ideas.

TermWhat it means for UGA Online
Repository (repo)A project folder on GitHub. UGA Online code lives under the uga-ool organization.
CloneCopy a repo from GitHub to your computer so you can edit files. You do this once per repo (or when you get a new machine).
main branchThe primary, shared version of the project. Do not commit directly to main for team code — use a branch and pull request.
BranchYour own line of work (e.g. feature/quiz-export). Safe place to experiment until review.
CommitA saved snapshot of your changes with a message describing why.
PushUpload your commits from your computer to GitHub.
Pull request (PR)Asking the team to merge your branch into main. Reviewers comment on GitHub before merge.
Pull / git pullDownload the latest changes from GitHub to your computer (do this often on main before starting new work).

Typical flow (what you will do)

  1. 1
    Clone the app repo (and this docs repo) to a folder on your Mac/PC.
  2. 2
    Create a branch from main for your task.
  3. 3
    Edit files in Cursor; review AI diffs before accepting.
  4. 4
    Commit & push your branch to GitHub.
  5. 5
    Open a PR on github.com; request review; address comments; merge when approved.
🌐
GitHub in the browser vs terminal You can create PRs and review entirely on github.com. Optional: install GitHub CLI (gh) for commands like gh pr checkout 42. Cursor’s terminal can run git for you when you ask — still review what it runs.

One-time setup on your computer

Follow these steps once. Your team lead can confirm org access and which repos you need.

1. Install tools

  • A
    Cursor — Download and sign in. On first launch, choose Import from VS Code if you already use VS Code (extensions and settings carry over).
  • B
    Git — macOS often has it; Windows: install Git for Windows. Verify: open Terminal and run git --version.
  • C
    GitHub access — Your UGA/GitHub account must be in the uga-ool org. Use HTTPS or SSH to clone (your lead can help with SSH keys).
  • D
    Node.js (for React apps) — Install LTS from nodejs.org if you will run npm install / npm run build.

2. Folder layout (recommended)

Keep sibling folders under one parent (many of us use local_dev/). Clone the template repo you work in first:

local_dev/ uga-lit-components/ ← Lit library (Cursor + commits here) uga-online-brightspace-agent-framework/ ← AI agents UGA-Brightspace-React-Apps/ ← React course-file apps uga-online-cursor-docs/ ← optional: topic docs + this page UGA-Brightspace-React-Template/ ← read-only reference (do not build new apps here)

3. Clone your template repo

Your lead will tell you which repo. Example (Lit library):

cd ~/local_dev git clone https://github.com/uga-ool/uga-lit-components.git

Optional — topic documentation hub:

git clone https://github.com/uga-ool/uga-online-cursor-docs.git

4. Open in Cursor (simplest)

✓
Recommended for beginners File → Open Folder… → select the template repo you cloned (e.g. uga-lit-components). Cursor loads that repo’s .cursor/ rules and commands automatically.
  1. 1
    Open one repo folder — The repo you commit to (Lit, agent framework, or React Apps).
  2. 2
    Read that repo’s guide — docs/cursor/README.md on GitHub or in your clone.
  3. 3
    Stay current — git pull in that repo when the team announces updates.
i
Multi-root workspace (optional) Advanced users may open several repos in one window via a .code-workspace file (workspaces/examples/ in cursor-docs). If two repos both have .cursor/, Cursor merges rules — prefer one repo when learning.

How to commit (beginners)

A commit saves your changes on your branch with a short note. Each template repo includes .github/COMMIT_TEMPLATE with hints.

  1. 1
    Save all files (⌘/Ctrl+S).
  2. 2
    Source Control — Click the branch icon on the left in Cursor.
  3. 3
    Stage the files you intend to commit — uncheck mistakes.
  4. 4
    Pre-commit review — Command Palette → pre-commit-review (in that template repo). Fix anything flagged; run again if needed.
  5. 5
    One short line in the message box (use the suggestion from the review) — examples:
    • Add uga-banner Lit component and demo
    • Add my-agent under frontend/src/agents/my-agent
    • Add my-tool under apps/my-tool
  6. 6
    Commit → then Push (upload to GitHub). Ask your lead the first time.
  7. 7
    Pull request on github.com when ready for review — see Pull requests. Before the PR, run hub command uga-online-pre-pr-check if your team uses this docs repo.

Full steps in your repo: docs/cursor/how-to-commit.md. After clone, run scripts/setup-git-hooks.sh once so commits use .github/COMMIT_TEMPLATE. Then run pre-commit-review before every commit.

How often? (team suggestion)

ActivitySuggested cadence
Local commitsEvery 30–90 minutes of meaningful progress
Push to remote branch1–3 times per day
Pull requestEvery 1–2 days
Merge to mainDaily or every few days (follow your lead)

5c

Secrets stay on your computer

Passwords, API keys, and .env files stay on your machine — instructional designers and developers do not commit them to GitHub.

Do

Use .env.example as a blank form; copy to .env locally and fill in real values there.

Do not

Commit .env, tokens, Kaltura admin keys, or LLM API keys. Do not paste them into Cursor chat.

FERPA

No student names, UGA IDs, or grades in commits or AI messages.

Team Rules and hub rules: uga-online-secrets-and-ferpa. Template repos may also include docs/cursor/secrets.md.

Pull requests & code review

    Changes are in the correct repo (apps under UGA-Brightspace-React-Apps/apps/).

Agent modes (Ask, Agent, Plan…)

Pick the mode in the chat panel before you prompt. When unsure, start with Ask.

Ask — read-only

Explains code; does not edit files. Safe for learning and PR review prep.

Agent — autonomous

Edits files, runs commands. Always review diffs. Default for implementation.

Plan — approve first

Writes a plan before changing code. Good for large or risky tasks.

Debug

Runtime investigation for bugs. Use when something fails in browser or Node.
🔁
Recommended pattern: Ask (scope) → Plan (optional, big tasks) → Agent (execute) → inline ⌘/Ctrl+K for small text tweaks.

11

Security & FERPA

Rules in uga-online-secrets-and-ferpa.mdc apply in every chat. Human judgment still required.

Never commit secrets

No API keys, OAuth secrets, or .env files. Use gitignored env files and institutional secret stores.

FERPA

Do not put student names, UGA IDs, or identifiable grades in prompts or committed files. Use aggregated anonymized data when appropriate.

Privacy Mode

Your org may require Cursor Privacy Mode — prompts not used for model training. Confirm with your admin.
